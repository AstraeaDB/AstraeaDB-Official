Chapter 15: Real-World Scenario — Cybersecurity Threat Investigation

This capstone chapter brings together nearly every feature covered in the guide—node and edge creation, graph traversals, GQL queries, vector search, GraphRAG, temporal analysis, and graph algorithms—in the context of a realistic cybersecurity threat investigation. You will build a threat graph from scratch, investigate a simulated breach, enrich findings with AI, and trace the attack timeline using temporal edges.

Prerequisites This chapter assumes you have worked through the earlier chapters and are comfortable with the AstraeaDB Python client. The primary examples are in Python; tabs for other languages are provided for key operations. Make sure your AstraeaDB server is running before following along.

15.1 Building the Threat Graph

The first step in any graph-powered security investigation is modeling the environment. In cybersecurity, entities and their relationships form a natural graph. Here is the data model we will build:

Node Types

LabelDescriptionExample Properties
Host A server, workstation, or network device hostname, os, ip, criticality
IPAddress An external or internal IP address address, geo_country, is_tor
Vulnerability A known CVE or weakness cve_id, severity, description, cvss_score
Alert A security alert from a SIEM or IDS alert_id, severity, rule_name, timestamp
AttackPattern A MITRE ATT&CK technique or tactic technique_id, name, tactic
User A user account (human or service) username, department, privilege_level

Edge Types

TypeFromToMeaning
CONNECTS_TOHostHostNetwork connectivity (with port and protocol)
HAS_VULNERABILITYHostVulnerabilityHost is affected by this CVE
TRIGGERED_ALERTHostAlertAn alert fired on this host
EXPLOITSAttackPatternVulnerabilityThis technique exploits this vulnerability
LOGGED_IN_FROMUserHostUser authenticated from this host
COMMUNICATES_WITHHostIPAddressOutbound connection to external IP

Building the Graph Step by Step

from astraeadb import AstraeaClient

with AstraeaClient("127.0.0.1", 7687, auth_token="admin-token") as client:

    # ── Hosts ──────────────────────────────────────────────
    web_server = client.create_node(
        ["Host"],
        {
            "hostname": "web-prod-01",
            "os": "Ubuntu 22.04",
            "ip": "10.0.1.10",
            "criticality": "high"
        },
        embedding=[0.12, 0.85, -0.34, 0.67, 0.91]  # From security description
    )
    print(f"Created web server: {web_server}")

    db_server = client.create_node(
        ["Host"],
        {
            "hostname": "db-prod-01",
            "os": "Ubuntu 22.04",
            "ip": "10.0.2.20",
            "criticality": "critical"
        },
        embedding=[0.08, 0.72, -0.41, 0.55, 0.88]
    )

    jumpbox = client.create_node(
        ["Host"],
        {
            "hostname": "jumpbox-01",
            "os": "Ubuntu 22.04",
            "ip": "10.0.0.5",
            "criticality": "high"
        }
    )

    # ── External IP (suspicious) ──────────────────────────
    external_ip = client.create_node(
        ["IPAddress"],
        {
            "address": "198.51.100.42",
            "geo_country": "Unknown",
            "is_tor": True
        }
    )

    # ── Vulnerability ─────────────────────────────────────
    cve = client.create_node(
        ["Vulnerability"],
        {
            "cve_id": "CVE-2024-1234",
            "severity": "Critical",
            "description": "Remote code execution in web framework",
            "cvss_score": 9.8
        }
    )

    # ── Alert ─────────────────────────────────────────────
    alert = client.create_node(
        ["Alert"],
        {
            "alert_id": "ALT-2024-5678",
            "severity": "Critical",
            "rule_name": "Reverse Shell Detected",
            "timestamp": "2024-11-15T03:22:00Z"
        }
    )

    # ── Attack Pattern (MITRE ATT&CK) ────────────────────
    attack_pattern = client.create_node(
        ["AttackPattern"],
        {
            "technique_id": "T1059.004",
            "name": "Unix Shell Command Execution",
            "tactic": "Execution"
        }
    )

    # ── User ──────────────────────────────────────────────
    admin_user = client.create_node(
        ["User"],
        {
            "username": "svc-deploy",
            "department": "Engineering",
            "privilege_level": "admin"
        }
    )

    # ── Edges (Relationships) ─────────────────────────────

    # Network topology
    client.create_edge(web_server, db_server, "CONNECTS_TO",
        {"port": 5432, "protocol": "TCP"})
    client.create_edge(jumpbox, web_server, "CONNECTS_TO",
        {"port": 22, "protocol": "SSH"})

    # Vulnerability exposure
    client.create_edge(web_server, cve, "HAS_VULNERABILITY")

    # Alert association
    client.create_edge(web_server, alert, "TRIGGERED_ALERT")

    # Attack pattern exploits vulnerability
    client.create_edge(attack_pattern, cve, "EXPLOITS")

    # Suspicious external communication
    client.create_edge(web_server, external_ip, "COMMUNICATES_WITH",
        {"bytes_sent": 524288, "first_seen": "2024-11-15T03:20:00Z"})

    # User login
    client.create_edge(admin_user, jumpbox, "LOGGED_IN_FROM",
        {"timestamp": "2024-11-15T03:15:00Z"})

    print("Threat graph built successfully.")

package main

import (
    "context"
    "fmt"
    "github.com/jimeharrisjr/astraeadb-go"
)

func main() {
    client := astraeadb.NewClient(
        astraeadb.WithAddress("127.0.0.1", 7687),
        astraeadb.WithAuthToken("admin-token"),
    )
    ctx := context.Background()
    client.Connect(ctx)
    defer client.Close()

    // Create hosts
    webServer, _ := client.CreateNode(ctx,
        []string{"Host"},
        map[string]any{
            "hostname":    "web-prod-01",
            "os":          "Ubuntu 22.04",
            "ip":          "10.0.1.10",
            "criticality": "high",
        },
        astraeadb.WithEmbedding([]float32{0.12, 0.85, -0.34, 0.67, 0.91}),
    )

    dbServer, _ := client.CreateNode(ctx,
        []string{"Host"},
        map[string]any{
            "hostname":    "db-prod-01",
            "os":          "Ubuntu 22.04",
            "ip":          "10.0.2.20",
            "criticality": "critical",
        },
    )

    cve, _ := client.CreateNode(ctx,
        []string{"Vulnerability"},
        map[string]any{
            "cve_id":      "CVE-2024-1234",
            "severity":    "Critical",
            "description": "Remote code execution in web framework",
        },
    )

    // Create edges
    client.CreateEdge(ctx, webServer, dbServer, "CONNECTS_TO",
        map[string]any{"port": 5432})
    client.CreateEdge(ctx, webServer, cve, "HAS_VULNERABILITY", nil)

    fmt.Println("Threat graph built successfully.")
}

15.2 Investigating an Alert

The scenario: at 03:22 UTC, a critical alert fires on web-prod-01 — a reverse shell has been detected. As the security analyst, your job is to determine the blast radius, trace the attack path, and identify what assets are at risk.

Step 1: Determine the Blast Radius with BFS

Use breadth-first search to find all nodes reachable from the compromised host within 3 hops. This tells you the potential impact of the breach. 

# How far can the attacker reach from the web server?
blast_radius = client.bfs(web_server, max_depth=3)
print(f"Potential impact: {len(blast_radius)} nodes within 3 hops")
print("Affected entities:")
for node in blast_radius:
    print(f"  - {node}")

Expected output:

Potential impact: 7 nodes within 3 hops
Affected entities:
  - web-prod-01 (Host)
  - db-prod-01 (Host)
  - CVE-2024-1234 (Vulnerability)
  - ALT-2024-5678 (Alert)
  - 198.51.100.42 (IPAddress)
  - jumpbox-01 (Host)
  - svc-deploy (User)
Critical Finding The database server db-prod-01 is within 1 hop of the compromised web server. If the attacker has code execution on web-prod-01, they can potentially reach the database via the PostgreSQL connection on port 5432. This is the highest-priority risk to investigate.

Step 2: Trace the Attack Path

Use shortest path to find how the attacker could have reached the database server from the external IP address: 

# How could the attacker reach the database?
path = client.shortest_path(external_ip, db_server)
print("Attack path:", " -> ".join(str(n) for n in path["path"]))
print(f"Path length: {path['distance']} hops")

Expected output:

Attack path: 198.51.100.42 -> web-prod-01 -> db-prod-01
Path length: 2 hops

Step 3: Find Related Vulnerabilities

Use a neighbor query to find all vulnerabilities associated with hosts in the blast radius: 

# What vulnerabilities exist on the compromised host?
neighbors = client.neighbors(web_server, direction="outgoing")
vulns = [n for n in neighbors if "Vulnerability" in n.get("labels", [])]
for v in vulns:
    print(f"  CVE: {v['cve_id']}  Severity: {v['severity']}  CVSS: {v['cvss_score']}")

Step 4: GQL Query for Targeted Investigation

Use a GQL query to find all critical vulnerabilities across the entire environment, along with the hosts they affect: 

# Find all critical vulnerabilities and their hosts
result = client.query(
    'MATCH (h:Host)-[:HAS_VULNERABILITY]->(v:Vulnerability) '
    'WHERE v.severity = "Critical" '
    'RETURN h.hostname, h.ip, v.cve_id, v.cvss_score '
    'ORDER BY v.cvss_score DESC'
)
print("Critical vulnerabilities in environment:")
for row in result:
    print(f"  {row['h.hostname']} ({row['h.ip']}): {row['v.cve_id']} (CVSS {row['v.cvss_score']})")

You can also trace which attack patterns exploit known vulnerabilities:

# Which MITRE ATT&CK techniques could be used against our hosts?
result = client.query(
    'MATCH (a:AttackPattern)-[:EXPLOITS]->(v:Vulnerability)'
    '      <-(h:Host) '
    'RETURN h.hostname, v.cve_id, a.technique_id, a.name'
)
for row in result:
    print(f"  {row['h.hostname']}: {row['a.technique_id']} - {row['a.name']}")

15.3 Enriching with AI

The graph gives us structure. AI gives us semantics. AstraeaDB's vector search and GraphRAG capabilities allow the analyst to go beyond exact matches and find similar threats, then generate human-readable briefings.

Vector Search: Find Similar Threats

If you have embedded the alert description into a vector (using any embedding model), you can find historically similar threats in the database: 

# Generate an embedding from the alert description
# (In practice, use an embedding model like OpenAI, Cohere, etc.)
alert_embedding = [0.15, 0.82, -0.31, 0.69, 0.87]

# Find the 5 most similar nodes in the graph
similar_threats = client.vector_search(alert_embedding, k=5)
print("Similar threats found:")
for threat in similar_threats:
    print(f"  - {threat['labels']}: {threat['properties']}")
    print(f"    Similarity: {threat['score']:.4f}")

Because AstraeaDB's vector index uses graph-based ANN (HNSW), the navigation links in the index are graph edges. This means you can perform hybrid search—finding nodes that are both semantically similar (close in vector space) and structurally connected (reachable via graph edges): 

# Hybrid search: similar AND connected
# "Find hosts within 2 hops of web-prod-01 that are most similar
#  to a 'database server with sensitive data' description"
results = client.vector_search(
    db_description_embedding,
    k=5,
    anchor=web_server,
    max_hops=2
)
for r in results:
    print(f"  {r['properties']['hostname']} (similarity: {r['score']:.4f}, hops: {r['distance']})")

GraphRAG: AI-Generated Analyst Briefing

The most powerful integration is GraphRAG (Chapter 11). AstraeaDB retrieves the relevant subgraph around the compromised host, linearizes it into text, and sends it to an LLM—all in a single atomic operation: 

# Generate an AI-powered analyst briefing
briefing = client.graph_rag(
    question="Summarize the threat landscape around web-prod-01 "
             "and recommend immediate actions.",
    anchor=web_server,
    hops=3,
    max_nodes=100,
    format="structured"
)

print("=== THREAT BRIEFING ===")
print(briefing["answer"])

Example output from the LLM:

=== THREAT BRIEFING ===

SITUATION SUMMARY
-----------------
Host web-prod-01 (10.0.1.10) triggered a critical alert (ALT-2024-5678:
"Reverse Shell Detected") at 2024-11-15T03:22:00Z. The host has a known
critical vulnerability (CVE-2024-1234, CVSS 9.8) — a remote code execution
flaw in its web framework.

The host is communicating with a suspicious external IP (198.51.100.42)
identified as a Tor exit node. 524 KB of data has been exfiltrated since
03:20 UTC.

BLAST RADIUS
-------------
- db-prod-01 (10.0.2.20): CRITICAL. Directly reachable via PostgreSQL
  (port 5432). Contains production data.
- jumpbox-01 (10.0.0.5): HIGH. SSH connectivity to web-prod-01.
  Compromise here grants lateral access to the management network.

RECOMMENDED ACTIONS
-------------------
1. IMMEDIATE: Isolate web-prod-01 from the network.
2. IMMEDIATE: Block outbound traffic to 198.51.100.42 at the firewall.
3. URGENT: Rotate credentials for svc-deploy (admin account logged into
   jumpbox-01 at 03:15 UTC — 7 minutes before the alert).
4. URGENT: Patch CVE-2024-1234 on web-prod-01 and scan all hosts for
   the same vulnerability.
5. INVESTIGATE: Analyze jumpbox-01 logs for signs of lateral movement.
Why GraphRAG beats plain RAG A traditional RAG system would retrieve documents similar to the question. GraphRAG retrieves the structural neighborhood of the affected entity. The LLM sees not just the alert, but the vulnerability it relates to, the network path to the database, the suspicious external IP, and the user who logged in minutes before the attack. This context produces dramatically more actionable briefings.

15.4 Temporal Analysis

Security investigations are inherently temporal. When did the vulnerability first appear? Was the suspicious connection established before or after the alert? Who logged in during the attack window? AstraeaDB's temporal edge support (Chapter 9) lets you answer these questions precisely.

Creating Temporal Edges

Edges can have validity intervals expressed as Unix timestamps (milliseconds). This records not just what happened, but when: 

# When was the vulnerability first discovered on this host?
client.create_edge(
    web_server, cve, "HAS_VULNERABILITY",
    valid_from=1700000000000    # Nov 14, 2023 — vulnerability first seen
)

# When did the suspicious outbound connection start?
client.create_edge(
    web_server, external_ip, "COMMUNICATES_WITH",
    {"bytes_sent": 524288},
    valid_from=1731642000000,   # Nov 15, 2024 03:20 UTC — connection started
    valid_to=1731645600000      # Nov 15, 2024 04:20 UTC — connection ended
)

# When did the user log in?
client.create_edge(
    admin_user, jumpbox, "LOGGED_IN_FROM",
    valid_from=1731641700000,   # Nov 15, 2024 03:15 UTC
    valid_to=1731649200000      # Nov 15, 2024 05:20 UTC
)

Querying the Graph at Different Points in Time

With temporal edges, you can ask "what did the graph look like at time T?" This is invaluable for reconstructing the attack timeline: 

# What was connected to web-prod-01 BEFORE the attack?
pre_attack = client.neighbors_at(
    web_server, "outgoing",
    timestamp=1731636000000    # Nov 15, 2024 01:00 UTC (2 hours before)
)
print("State before attack:")
for n in pre_attack:
    print(f"  - {n}")

# What was connected DURING the attack window?
during_attack = client.neighbors_at(
    web_server, "outgoing",
    timestamp=1731642600000    # Nov 15, 2024 03:30 UTC (during the attack)
)
print("\nState during attack:")
for n in during_attack:
    print(f"  - {n}")

Expected output:

State before attack:
  - db-prod-01 (CONNECTS_TO, port 5432)
  - CVE-2024-1234 (HAS_VULNERABILITY)

State during attack:
  - db-prod-01 (CONNECTS_TO, port 5432)
  - CVE-2024-1234 (HAS_VULNERABILITY)
  - 198.51.100.42 (COMMUNICATES_WITH)          <-- NEW: suspicious connection
  - ALT-2024-5678 (TRIGGERED_ALERT)            <-- NEW: alert fired

Temporal Shortest Path

You can also compute shortest paths that only use edges valid at a specific time. This reveals the actual attack path as it existed during the incident: 

# What was the shortest path from external IP to database
# at the time of the attack?
temporal_path = client.shortest_path(
    external_ip, db_server,
    timestamp=1731642600000    # During the attack
)
print("Attack path at time of incident:")
print(" -> ".join(str(n) for n in temporal_path["path"]))

Reconstructing the Timeline

Putting it all together, we can reconstruct the complete attack timeline:

Time (UTC)EventEvidence in Graph
Nov 2023 CVE-2024-1234 first affects web-prod-01 HAS_VULNERABILITY edge valid_from
03:15 svc-deploy logs into jumpbox-01 LOGGED_IN_FROM edge valid_from
03:20 web-prod-01 begins communicating with Tor exit node COMMUNICATES_WITH edge valid_from
03:22 Reverse shell alert fires on web-prod-01 TRIGGERED_ALERT edge + Alert node timestamp
04:20 Outbound connection to Tor node ends COMMUNICATES_WITH edge valid_to
Key Observation The svc-deploy login at 03:15 UTC—7 minutes before the reverse shell alert—is suspicious. Was this a legitimate deployment, or did the attacker compromise the service account and use it to access the jumpbox? This is exactly the kind of temporal correlation that graph databases excel at revealing.

15.5 Bringing It All Together

In this scenario, we used the following AstraeaDB features to investigate a cybersecurity breach:

FeatureChapterHow We Used It
Node and Edge creation Ch 4 Built the threat graph with hosts, IPs, vulnerabilities, alerts, and users
BFS (Breadth-First Search) Ch 10 Determined the blast radius of the compromised host
Shortest Path Ch 10 Traced the attack path from external IP to database server
Neighbor Queries Ch 6 Found vulnerabilities and alerts associated with the compromised host
GQL Queries Ch 5 Pattern-matched across the graph for critical vulnerabilities and ATT&CK techniques
Vector Search Ch 8 Found historically similar threats using embedding similarity
Hybrid Search Ch 8 Combined structural proximity with semantic similarity
GraphRAG Ch 11 Generated an AI-powered analyst briefing with actionable recommendations
Temporal Edges Ch 9 Recorded when events occurred and queried the graph at specific points in time
Temporal Shortest Path Ch 9 Computed the attack path as it existed during the incident window
Authentication (RBAC) Ch 13 Connected with appropriate credentials for the investigation

This scenario demonstrates why graph databases are uniquely suited for cybersecurity. The relationships between entities—network connections, vulnerability exposure, user logins, alert associations—are the core data. Relational databases would require dozens of JOIN operations to trace the attack path. AstraeaDB follows pointers directly, making multi-hop investigations interactive rather than batch operations.

The addition of vector embeddings and GraphRAG transforms AstraeaDB from a data store into an investigative partner. Instead of an analyst manually exploring each connection, the system can surface similar historical threats, correlate temporal patterns, and generate briefings that synthesize the entire neighborhood of the incident.

Where to go from here This chapter only scratches the surface. Consider extending the threat graph with additional data sources: firewall logs (as temporal edges), DNS resolution records, file hash observations, and threat intelligence feeds. The richer the graph, the more powerful the investigation.
